The Wizzx API uses conventional HTTP response codes to indicate the success or failure of an API request. The API returns errors in an OpenAI-compatible format for easy integration with existing SDKs.

## Error Response Format

All error responses follow this structure:

```json
{
  "error": {
    "message": "Human-readable error message with solution",
    "type": "error_category",
    "param": "parameter_name",
    "code": "specific_error_code"
  },
  "request_id": "req_abc123"
}
```

| Field | Description |
|-------|-------------|
| `message` | A human-readable description of the error with suggested solutions |
| `type` | The category of error, useful for SDK exception mapping |
| `param` | The parameter that caused the error (null if not applicable) |
| `code` | A specific error code for programmatic handling |
| `request_id` | Unique identifier for the request, useful for support |

## HTTP Status Codes Overview

| Status Code | Type | Description |
|-------------|------|-------------|
| 400 | `invalid_request_error` | The request was malformed or missing required parameters |
| 401 | `authentication_error` | Invalid or missing API key |
| 402 | `insufficient_quota` | Not enough credits to complete the request |
| 403 | `permission_denied` | You don't have access to the requested resource |
| 404 | `not_found_error` | The requested resource doesn't exist |
| 409 | `conflict_error` | The request conflicts with current state |
| 422 | `unprocessable_entity_error` | The request was well-formed but cannot be processed |
| 429 | `rate_limit_error` | Too many requests in a short period |
| 500 | `server_error` | Something went wrong on our end |
| 503 | `service_unavailable` | The service is temporarily unavailable |

## Authentication Errors (401)

Authentication errors occur when there's an issue with your API key.

### `missing_api_key`

```json
{
  "error": {
    "message": "Missing API key. Include your API key in the Authorization header using Bearer authentication.",
    "type": "authentication_error",
    "param": null,
    "code": "missing_api_key"
  }
}
```

**Solution:** Add the `Authorization` header to your request:

```bash
curl -H "Authorization: Bearer YOUR_API_KEY" ...
```

### `invalid_api_key`

```json
{
  "error": {
    "message": "Invalid API key provided. Ensure the API key is correct and active.",
    "type": "authentication_error",
    "param": null,
    "code": "invalid_api_key"
  }
}
```

**Solution:** Verify your API key is correct. You can find your API keys in the [Dashboard](https://dashboard.wizzx.ai).

### `api_key_revoked`

```json
{
  "error": {
    "message": "API key has been revoked. Generate a new API key from your dashboard.",
    "type": "authentication_error",
    "param": null,
    "code": "api_key_revoked"
  }
}
```

**Solution:** Generate a new API key from your dashboard.

    "param": null,
    "code": "invalid_json_body"

### `resource_conflict`

```json

    "type": "conflict_error",
    "param": null,
    "code": "resource_conflict"
  }
}
```

**Solution:** Refresh your local state and retry the operation.

### `duplicate_resource`

```json
{
  "error": {
    "message": "Resource 'my-api-key' already exists. Use a different identifier or update the existing resource.",
    "type": "conflict_error",
    "param": null,
    "code": "duplicate_resource"
  }
}
```

**Solution:** Use a unique identifier or update the existing resource instead.

### `concurrent_update`

```json
{
  "error": {
    "message": "The resource was modified by another request. Refresh and retry your update.",
    "type": "conflict_error",
    "param": null,
    "code": "concurrent_update"
  }
}
```

**Solution:** Fetch the latest version and retry your update.

## Rate Limit Errors (429)

These errors indicate you're sending requests too quickly.

### `rate_limit_exceeded`

```json
{
  "error": {
    "message": "Rate limit exceeded. You are sending requests too quickly. Retry after 60 seconds.",
    "type": "rate_limit_error",
    "param": null,
    "code": "rate_limit_exceeded"
  },
  "retry_after": 60
}
```

**Solution:** Implement exponential backoff in your code. The `retry_after` field indicates when you can retry.

```python
import time

def call_api_with_retry(func, max_retries=3):
    for attempt in range(max_retries):
        try:
            return func()
        except RateLimitError as e:
            if attempt < max_retries - 1:
                time.sleep(e.retry_after or 60)
            else:
                raise
```

### `requests_quota_exceeded`

```json
{
  "error": {
    "message": "You exceeded your request quota. Wait until your quota resets or upgrade your plan.",
    "type": "rate_limit_error",
    "param": null,
    "code": "requests_quota_exceeded"
  }
}
```

**Solution:** Wait for your quota to reset or upgrade to a higher tier.

## Error Handling Best Practices

<AccordionGroup>
  <Accordion title="Implement Retry Logic">
    Always implement retry logic with exponential backoff for transient errors (429, 500, 503):

    ```python
    import time
    import random

    def retry_with_backoff(func, max_retries=3, base_delay=1):
        for attempt in range(max_retries):
            try:
                return func()
            except (RateLimitError, ServerError) as e:
                if attempt == max_retries - 1:
                    raise
                delay = base_delay * (2 ** attempt) + random.uniform(0, 1)
                time.sleep(delay)
    ```
  </Accordion>

  <Accordion title="Log Request IDs">
    Always log the `request_id` from error responses. This helps our support team diagnose issues quickly.
  </Accordion>

  <Accordion title="Handle Specific Error Codes">

    ```python
    try:
        response = api.generate_video(...)
    except APIError as e:
        if e.code == "insufficient_credits":
            notify_user_to_top_up()
        elif e.code == "rate_limit_exceeded":
            retry_later(e.retry_after)
        else:
            log_error(e.request_id)
    ```
  </Accordion>
